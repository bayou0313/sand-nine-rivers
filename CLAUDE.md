# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm run dev              # Vite dev server on :8080 (HMR overlay disabled)
npm run build            # Production build → dist/
npm run build:dev        # Build with mode=development (keeps lovable-tagger)
npm run lint             # ESLint (flat config, react-hooks + react-refresh)
npm run test             # Vitest, run-once
npm run test:watch       # Vitest watch
npx vitest run src/test/example.test.ts   # Run a single test file
npx playwright test      # E2E (uses lovable-agent-playwright-config)
node scripts/prerender-cities.mjs         # Manually regenerate city SEO shells after build
```

The `postbuild` hook runs the city pre-render automatically after `npm run build`.

Env vars (`.env`): `VITE_SUPABASE_URL`, `VITE_SUPABASE_PUBLISHABLE_KEY`, `VITE_SUPABASE_PROJECT_ID`. The `SUPABASE_URL` / `SUPABASE_ANON_KEY` (no `VITE_` prefix) are read by the pre-render script only.

## Architecture

**Stack:** Vite + React 18 + TypeScript SPA, React Router v6, Tailwind + shadcn/ui (Radix), TanStack Query, react-hook-form + zod, Supabase (Postgres + Edge Functions + Auth + Storage), Stripe, Google Maps (Places / Geocoding / Distance Matrix), Resend for email, GTM + Microsoft Clarity for analytics.

**Deploy:** `main` → GitHub Actions (`.github/workflows/deploy.yml`) builds, runs `prerender-cities.mjs`, pushes `dist/` to `gh-pages` with `CNAME=riversand.net`. `vercel.json` proxies `/sitemap.xml` to the `generate-sitemap` edge function and falls back all other routes to `/` (SPA).

### Routing is mobile/desktop split at the route level

`src/App.tsx` defines `HomeRouter` and `OrderRouter` which pick between desktop (`Index.tsx` / `Order.tsx`) and mobile (`HomeMobile.tsx` / `OrderMobile.tsx`) variants via `useIsMobile()`. `localStorage.force_desktop === "true"` forces the desktop variant. **Changes to homepage or order flow must usually be made in both variants.**

### Global site/payment modes gate the app

On mount, `AppContent` reads two `global_settings` rows:
- `site_mode`: `"maintenance"` renders a full-screen maintenance page on all non-admin routes (bypass via `sessionStorage.maintenance_bypass`).
- `stripe_mode`: `"test"` shows a persistent orange banner + dismissible modal on non-admin routes. Tests do not charge.

`--banner-offset` CSS var is set from this and the Navbar shifts down accordingly.

### Two separate admin systems

1. **`/admin`** — Supabase Auth (`auth.signInWithPassword`) + `user_roles.role = 'admin'` check. Order management only. Direct Supabase table reads/writes.
2. **`/leads`** — password-gated via the `leads-auth` edge function using the `LEADS_PASSWORD` secret. All data access goes through that edge function (one action verb per call: `list`, `save_pit`, `generate_city_content`, etc.). This is the main operator dashboard (~4200 lines in `Leads.tsx`).

Don't conflate the two. When adding admin features, pick the dashboard by the auth model you need.

### City SEO pages and pre-rendering

`/:citySlug/river-sand-delivery` dynamically renders from the `city_pages` table. For SEO, `scripts/prerender-cities.mjs` runs after build: it fetches each active city from Supabase REST and writes a custom `dist/<slug>/river-sand-delivery/index.html` with city-specific `<head>` (title, meta, canonical, OG, JSON-LD) plus the Vite asset tags. React then hydrates into the empty `#root`. Legacy `<slug>-la/...` URLs are both client-side redirected (in `App.tsx`) **and** written as static redirect HTML by the pre-render script.

Content for city pages is generated by the `generate-city-page` edge function using Anthropic Claude (`ANTHROPIC_API_KEY`).

### Data boundary: Supabase

- Client: `src/integrations/supabase/client.ts` (auto-generated — do not edit). Typed via `Database` from `types.ts`.
- Direct table reads from the app: `global_settings`, `pits`, `city_pages`, `orders`, `payment_events`, `user_roles`, `delivery_leads`. Many call-sites cast `(supabase as any)` to bypass strict table types — this is a known debt area, not a convention to extend.
- RPCs in use: `increment_city_page_views(p_slug)`.
- Migrations live in `supabase/migrations/`. `DATABASE_SCHEMA.sql` at repo root is a consolidated snapshot (not executed — reference only).

### Edge Functions (`supabase/functions/`)

Most require auth (JWT). `send-email` has `verify_jwt = false` in `supabase/config.toml` because it's invoked from public surfaces (contact form, lead capture). Function inventory:

- **Payments:** `create-checkout-link`, `create-payment-intent`, `stripe-webhook`, `capture-payments`, `create-refund`
- **Orders:** `get-order-status`, `generate-invoice`
- **Email:** `send-email` (types: `order_confirmation`, `contact`, `out_of_area_lead`, `callback`, `proposal`, `cash_payment_confirmation`), `abandonment-emails`, `email-inbound`
- **SEO:** `generate-sitemap`, `submit-sitemap`, `generate-city-page`, `generate-docs`
- **Ops:** `leads-auth` (the admin dashboard gateway), `get-maps-key` (server-issued Google Maps browser key)

### Order flow (`Order.tsx` / `OrderMobile.tsx`)

Four steps: address → details → confirm → success. Reads `global_settings` for pricing + `pits` for active PITs, computes distance/price via Google Distance Matrix, matches customer coords to nearest PIT. Parish-based tax rate via `src/lib/format.ts` (rates currently hardcoded — not yet in `global_settings`). Card payments add a 3.5% processing fee. Stripe checkout return is handled both same-tab and cross-tab (localStorage signal). URL params drive most of the state (`address`, `distance`, `price`, `qty`, `discount`, `lead`, `utm_source`, `payment`, `order_number`, `session_id`, `return_mode`).

## Conventions

- **Path alias** `@/*` → `./src/*` (configured in `vite.config.ts`, `tsconfig.app.json`, `components.json`, `vitest.config.ts`).
- **Fonts:** `font-display` = Bebas Neue (uppercase headings), `font-body` / `font-sans` = Inter. Never DM Sans.
- **Colors:** app uses HSL CSS variables from `src/index.css` via Tailwind tokens. **The `/leads` LMT dashboard has its own design system** in `.lovable/GUIDELINES.md` with hardcoded brand hex (`BRAND_NAVY = #0D2137`, `BRAND_GOLD = #C07A00`) and a strict `T.*` token object — follow that file closely when touching `Leads.tsx` tabs.
- **Analytics:** all events go through `trackEvent` in `src/lib/analytics.ts` (GTM dataLayer). GTM is hardcoded in `index.html`; Clarity ID is pulled from `global_settings.seo_clarity_id` at runtime (skipped on `/admin` and `/leads`).
- **Toasts:** prefer `useToast` (`src/hooks/use-toast.ts`) for shadcn Toaster, or sonner for the LMT dashboard.
- Don't add `README.md`-style docs inside `src/`. The authoritative references are:
  - `COMPONENT_INVENTORY.md` — every page/component with props, supabase queries, edge functions, known debt. Consult before large refactors.
  - `DATABASE_SCHEMA.sql` — full schema snapshot (enums, tables, RLS, functions).
  - `.lovable/GUIDELINES.md` — LMT dashboard UI rules.

## Gotchas

- `(supabase as any)` casts are used to reach tables missing from generated types — check `src/integrations/supabase/types.ts` before "fixing" one, it's often easier to regenerate types than to untangle a cast.
- The Google Maps script is loaded independently by `DeliveryEstimator.tsx` and `Order.tsx` — if you're touching Maps loading, check both.
- Phone number `1-855-468-9297` (aka `1-855-GOT-WAYS`) is hardcoded in several components; don't assume a single source of truth.
- After adding new custom routes, place them **above** the `*` catch-all in `App.tsx` (comment marks the spot).
- When adding a new city slug, also add a legacy `<slug>-la` redirect in `App.tsx` **and** in `LEGACY_REDIRECTS` in `scripts/prerender-cities.mjs` if the city previously shipped under the `-la` suffix.

## Known operational rules not obvious from code

- `zip_tax_rates` queries require `.limit(10000)` — Supabase's default row limit is too low and silently truncates results. If tax lookups fail for certain ZIPs, this is usually why.
- When iterating on `/leads` features via Lovable, embed exact TypeScript template literals inline in the prompt rather than attaching files. Attachments produce worse results than inline code fences for this codebase.
- GTM container for riversand.net is GTM-KPKFPCXM (8 GA4 tags, 9 data layer variables). Separate containers exist for Sand N Soil (GTM-N6HV3NKF, G-71R1JNJG1K) and WAYS Materials (GTM-T9M8ZV8K, G-5618CP72HC), all under Haulogix Ads Central MCC 622-328-2005.
- Parish tax rates are hardcoded in `src/lib/format.ts` — moving them to `global_settings` is a known pending task.
- Google Ads account creation for WAYS is currently blocked by an unresolved policy violation. Do not assume a WAYS Google Ads account exists.

## Commit and PR conventions

- Main branch is `main`, deploys automatically on push.
- Before committing: run `npm run lint` and `npm run test` locally. CI will block on failures.
- For schema changes, add a migration to `supabase/migrations/` — do NOT edit `DATABASE_SCHEMA.sql` directly (it's a snapshot, not executed).
